## What does enforcement buy?

- **enforced** runtime: forbidden-crossing ASR = 0 on every journey: **YES** — the blocked crossing never applies, so `_forbidden_crossed` sees no applied transition on a target.
- **detect-only** runtime: the identical crossing flips ASR to 1 on every journey: **YES** — recording-but-applying lets the hijack land, and the oracle catches it.
- **canary egress is enforcement-independent**: it depends on what the agent writes to memory / sent messages / dream journal, not on whether a crossing is blocked. Enforcement protects the *crossing* path; agent behaviour protects the *egress* path. The two ASR mechanisms are orthogonal (security oracle §2).

## Conclusion

Runtime enforcement is **load-bearing** for the forbidden-crossing half of axis07: remove it (detect-only) and every goal-hijack succeeds (ASR 0 -> 1). The reported ASR-0 of real models is therefore a *genuine* resistance number only because the runtime actually blocks — exactly why the benchmark must report security from the enforcing stepwise runtime and label any detect-only numbers as such (plan AB4; AgentDojo with/without-defense framing, arXiv 2406.13352).
